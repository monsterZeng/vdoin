Intro to smart pointers and move semantics

# Intro to smart pointers and move semantics

考虑一个动态分配值的函数:

void someFunction()
{
    Resource *ptr = new Resource(); // Resource is a struct or class
 
    // do stuff with ptr here
 
    delete ptr;
}

在上述两个程序中，提前执行return或throw语句，导致函数在不删除变量ptr的情况下终止。因此，分配给变量ptr的内存现在被泄漏了(并且将在每次调用此函数并提前返回时再次泄漏)。

从本质上讲，这些问题的发生是因为指针变量没有内在的机制来自行清理。

# Smart pointer classes to the rescue?

类最好的一点是，它们包含析构函数，当类的对象超出作用域时，析构函数会自动执行。因此，如果你在构造函数中分配(或获取)内存，你可以在析构函数中释放它，并保证当类对象被销毁时(不管它是否超出作用域，被显式删除，等等)，内存将被释放。这是我们在12.9课中所讨论的RAII编程范式的核心——析构函数 (opens new window)。

那么，我们可以使用类来帮助我们管理和清理指针吗?我们可以!

考虑这样一个类，它的唯一工作是保存并“拥有”传递给它的指针，然后在类对象超出作用域时释放该指针。只要该类的对象只是作为局部变量创建的，我们就可以保证类将正确地超出作用域(不管函数何时或如何终止)，并且拥有的指针将被撤销。

以下是这个想法的初稿:

#include <iostream>
 
template<class T>
class Auto_ptr1
{
	T* m_ptr;
public:
	// Pass in a pointer to "own" via the constructor
	Auto_ptr1(T* ptr=nullptr)
		:m_ptr(ptr)
	{
	}
	
	// The destructor will make sure it gets deallocated
	~Auto_ptr1()
	{
		delete m_ptr;
	}
 
	// Overload dereference and operator-> so we can use Auto_ptr1 like m_ptr.
	T& operator*() const { return *m_ptr; }
	T* operator->() const { return m_ptr; }
};
 
// A sample class to prove the above works
class Resource
{
public:
    Resource() { std::cout << "Resource acquired\n"; }
    ~Resource() { std::cout << "Resource destroyed\n"; }
};
 
int main()
{
	Auto_ptr1<Resource> res(new Resource()); // Note the allocation of memory here
 
        // ... but no explicit delete needed
 
	// Also note that the Resource in angled braces doesn't need a * symbol, since that's supplied by the template
 
	return 0;
} // res goes out of scope here, and destroys the allocated Resource for us
/*
Resource acquired
Resource destroyed
*/

考虑一下这个程序和类是如何工作的。首先，我们动态创建一个Resource，并将它作为参数传递给模板化的Auto_ptr1类。从那以后，我们的Auto_ptr1变量res拥有这个Resource对象(Auto_ptr1与m_ptr有一个组合关系)。因为res被声明为局部变量并且具有块作用域，所以当块结束时，它将超出作用域，并被销毁(不用担心忘记释放它)。因为它是一个类，所以当destory它时，Auto_ptr1析构函数将被调用。该析构函数将确保它所持有的Resource指针被删除!

只要Auto_ptr1被定义为一个局部变量(具有自动持续时间，因此类名的“Auto”部分)，Resource将保证在声明它的块的末尾被销毁，而不管函数如何终止(即使它提前终止)。

这样的类称为智能指针。Smart指针是一个组合类，用于管理动态分配的内存，并确保在智能指针对象超出作用域时删除内存。(与此相关的是，内置指针有时被称为“哑指针”，因为它们自己无法清理)。

现在让我们回到上面的someFunction()例子，展示智能指针类如何解决我们的挑战:

#include <iostream>
 
template<class T>
class Auto_ptr1
{
	T* m_ptr;
public:
	// Pass in a pointer to "own" via the constructor
	Auto_ptr1(T* ptr=nullptr)
		:m_ptr(ptr)
	{
	}
	
	// The destructor will make sure it gets deallocated
	~Auto_ptr1()
	{
		delete m_ptr;
	}
 
	// Overload dereference and operator-> so we can use Auto_ptr1 like m_ptr.
	T& operator*() const { return *m_ptr; }
	T* operator->() const { return m_ptr; }
};
 
// A sample class to prove the above works
class Resource
{
public:
    Resource() { std::cout << "Resource acquired\n"; }
    ~Resource() { std::cout << "Resource destroyed\n"; }
    void sayHi() { std::cout << "Hi!\n"; }
};
 
void someFunction()
{
    Auto_ptr1<Resource> ptr(new Resource()); // ptr now owns the Resource
 
    int x;
    std::cout << "Enter an integer: ";
    std::cin >> x;
 
    if (x == 0)
        return; // the function returns early
 
    // do stuff with ptr here
    ptr->sayHi();
}
 
int main()
{
    someFunction();
 
    return 0;
}

# A critical flaw

Auto_ptr1类有一个隐藏在一些自动生成代码后面的关键缺陷。在继续阅读之前，看看你能不能确定它是什么。我们会等待……

我们不会告诉你，而是展示给你看。考虑以下程序:

include <iostream>
 
// Same as above
template<class T>
class Auto_ptr1
{
	T* m_ptr;
public:
	Auto_ptr1(T* ptr=nullptr)
		:m_ptr(ptr)
	{
	}
	
	~Auto_ptr1()
	{
		delete m_ptr;
	}
 
	T& operator*() const { return *m_ptr; }
	T* operator->() const { return m_ptr; }
};
 
class Resource
{
public:
	Resource() { std::cout << "Resource acquired\n"; }
	~Resource() { std::cout << "Resource destroyed\n"; }
};
 
int main()
{
	Auto_ptr1<Resource> res1(new Resource());
	Auto_ptr1<Resource> res2(res1); // Alternatively, don't initialize res2 and then assign res2 = res1;
 
	return 0;
}

很有可能(但不一定)您的程序将在此时崩溃。现在看到问题了吗?因为我们没有提供复制构造函数或赋值操作符，c++为我们提供了一个。它提供的函数做浅拷贝。因此，当我们用res1初始化res2时，两个Auto_ptr1变量都指向同一个Resource。当res2超出作用域时，它删除资源，留给res1一个悬空指针。当res1去删除它的(已经删除的)资源时，崩溃!

很明显这不是好事。我们如何解决这个问题?

我们可以做的一件事是显式地定义并删除复制构造函数和赋值操作符，从而在一开始就防止复制。这将阻止按值传递的情况(这很好，我们可能不应该按值传递这些)。

我们不能通过引用返回Auto_ptr1，因为本地的Auto_ptr1将在函数结束时被销毁，而调用者将留下一个悬空引用。按地址返回也有同样的问题。我们可以按地址返回指针r，但之后可能会忘记删除r，这就是最初使用智能指针的全部意义所在。这就是。按值返回Auto_ptr1是唯一有意义的选项——但随后我们会以浅拷贝、复制指针和崩溃告终。

另一种选择是重写复制构造函数和赋值操作符来创建深层复制。这样，我们至少可以保证避免重复指向同一个对象的指针。但是复制可能代价昂贵(可能不理想，甚至不可能)，而且我们不想仅仅为了从函数返回Auto_ptr1而对对象进行不必要的复制。另外，赋值或初始化一个笨指针不会复制被指向的对象，那么为什么我们会期望智能指针的行为不同呢?

What do we do?

# Move semantics

如果我们不使用复制构造函数和赋值操作符复制指针(“复制语义”)，而是将指针的所有权从源对象转移到目标对象，那会怎么样?这是移动语义背后的核心思想。Move语义意味着类将转移对象的所有权，而不是创建一个副本。

让我们更新Auto_ptr1类来演示如何做到这一点:

#include <iostream>
 
template<class T>
class Auto_ptr2
{
	T* m_ptr;
public:
	Auto_ptr2(T* ptr=nullptr)
		:m_ptr(ptr)
	{
	}
	
	~Auto_ptr2()
	{
		delete m_ptr;
	}
 
	// A copy constructor that implements move semantics
	Auto_ptr2(Auto_ptr2& a) // note: not const
	{
		m_ptr = a.m_ptr; // transfer our dumb pointer from the source to our local object
		a.m_ptr = nullptr; // make sure the source no longer owns the pointer
	}
	
	// An assignment operator that implements move semantics
	Auto_ptr2& operator=(Auto_ptr2& a) // note: not const
	{
		if (&a == this)
			return *this;
 
		delete m_ptr; // make sure we deallocate any pointer the destination is already holding first
		m_ptr = a.m_ptr; // then transfer our dumb pointer from the source to the local object
		a.m_ptr = nullptr; // make sure the source no longer owns the pointer
		return *this;
	}
 
	T& operator*() const { return *m_ptr; }
	T* operator->() const { return m_ptr; }
	bool isNull() const { return m_ptr == nullptr;  }
};
 
class Resource
{
public:
	Resource() { std::cout << "Resource acquired\n"; }
	~Resource() { std::cout << "Resource destroyed\n"; }
};
 
int main()
{
	Auto_ptr2<Resource> res1(new Resource());
	Auto_ptr2<Resource> res2; // Start as nullptr
 
	std::cout << "res1 is " << (res1.isNull() ? "null\n" : "not null\n");
	std::cout << "res2 is " << (res2.isNull() ? "null\n" : "not null\n");
 
	res2 = res1; // res2 assumes ownership, res1 is set to null
 
	std::cout << "Ownership transferred\n";
 
	std::cout << "res1 is " << (res1.isNull() ? "null\n" : "not null\n");
	std::cout << "res2 is " << (res2.isNull() ? "null\n" : "not null\n");
 
	return 0;
}

# std::auto_ptr, and why it was a bad idea

现在是讨论std::auto_ptr的恰当时机。在c++ 98中引入并在c++ 17中移除的std::auto_ptr，是c++对标准化智能指针的第一次尝试。auto_ptr选择像Auto_ptr2类那样实现move语义。

然而，std::auto_ptr(以及我们的Auto_ptr2类)有许多问题，这使得使用它很危险。

首先，因为std::auto_ptr通过复制构造函数和赋值操作符实现了move语义，所以按值向函数传递std::auto_ptr将导致将资源移动到函数形参中(并在函数形参超出作用域时在函数结束时销毁)。然后，当您从调用者访问您的auto_ptr参数时(没有意识到它已被转移和删除)，您突然取消了对空指针的引用。崩溃!

第二，std::auto_ptr总是使用非数组删除来删除它的内容。这意味着auto_ptr不能正确地与动态分配的数组一起工作，因为它使用了错误的释放类型。更糟糕的是，它不能阻止您向它传递动态数组，然后它会管理不当，导致内存泄漏。

最后，auto_ptr不能很好地与标准库中的许多其他类兼容，包括大多数容器和算法。这是因为那些标准库类假定当它们复制一个项时，实际上是复制，而不是移动。

由于上面提到的缺点，std::auto_ptr在c++ 11中已被弃用，在c++ 17中已被移除。

# Moving forward

auto_ptr设计的核心问题是在c++ 11之前，c++语言根本没有区分“复制语义”和“移动语义”的机制。覆盖复制语义来实现移动语义会导致奇怪的边缘情况和无意的bug。例如，您可以写入res1 = res2，但不知道res2是否会被更改!

因为如此，c++ 11正式定义了“移动”的概念，并在语言中添加了“移动语义”，以正确区分复制和移动。既然我们已经为移动语义是有用的做好了准备，我们将在本章的其余部分探讨移动语义的主题。我们还将使用move语义修复Auto_ptr2类。

在c++ 11中，std::auto_ptr已经被一堆其他类型的“move-aware”智能指针所取代:std::unique_ptr、std::weak_ptr和std::shared_ptr。我们还将探讨其中最流行的两个:unique_ptr(它是auto_ptr的直接替代品)和shared_ptr。
