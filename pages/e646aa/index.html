Pure virtual functions, abstract base classes, and interface classes

# Pure virtual functions, abstract base classes, and interface classes

# Pure virtual (abstract) functions and abstract base classes

到目前为止，我们编写的所有虚函数都有一个函数体(定义)。然而，c++允许您创建一种特殊的虚函数，称为纯虚函数(或抽象函数)，它根本没有主体（没有定义）!纯虚函数只是作为一个占位符，可以由派生类重新定义。

要创建纯虚函数，无需定义函数体，只需将值赋给函数0。

class Base
{
public:
    const char* sayHi() const { return "Hi"; } // a normal non-virtual function    
 
    virtual const char* getName() const { return "Base"; } // a normal virtual function
 
    virtual int getValue() const = 0; // a pure virtual function
 
    int doSomething() = 0; // Compile error: can not set non-virtual functions to 0
};

当我们在类中添加纯虚函数时，我们实际上是在说，“这取决于派生类来实现这个函数”。

使用纯虚函数有两个主要后果:首先，任何具有一个或多个纯虚函数的类都会成为抽象基类，这意味着它不能被实例化!考虑一下如果我们创建Base的一个实例会发生什么:

int main()
{
    Base base; // We can't instantiate an abstract base class, but for the sake of example, pretend this was allowed
    base.getValue(); // what would this do?
 
    return 0;
}

其次，任何派生类都必须为该函数定义一个主体，否则该派生类也将被视为抽象基类。

A pure virtual function example

让我们看一个纯虚函数的例子。在前一课中，我们编写了一个简单的Animal基类，并从中派生了一个Cat和Dog类。下面是我们离开时的代码:

#include <string>
#include <utility>
 
class Animal
{
protected:
    std::string m_name;
 
    // We're making this constructor protected because
    // we don't want people creating Animal objects directly,
    // but we still want derived classes to be able to use it.
    Animal(const std::string& name)
        : m_name{ name }
    {
    }
 
public:
    std::string getName() const { return m_name; }
    virtual const char* speak() const { return "???"; }
    
    virtual ~Animal() = default;
};
 
class Cat: public Animal
{
public:
    Cat(const std::string& name)
        : Animal{ name }
    {
    }
 
    const char* speak() const override { return "Meow"; }
};
 
class Dog: public Animal
{
public:
    Dog(const std::string& name)
        : Animal{ name }
    {
    }
 
    const char* speak() const override { return "Woof"; }
};

发生了什么事?我们忘了重新定义函数speak()，所以cow.Speak()决定改为Animal.speak()，这不是我们想要的。

更好的解决方案是使用纯虚函数:

#include <string>
 
class Animal // This Animal is an abstract base class
{
protected:
    std::string m_name;
 
public:
    Animal(const std::string& name)
        : m_name{ name }
    {
    }
 
    const std::string& getName() const { return m_name; }
    virtual const char* speak() const = 0; // note that speak is now a pure virtual function
    
    virtual ~Animal() = default;
};

当我们想将一个函数放入基类中，但只有派生类知道它应该返回什么时，纯虚函数是有用的。纯虚函数使基类无法实例化，派生类必须在实例化之前定义这些函数。这有助于确保派生类不会忘记重定义基类期望它们去做的函数。

Pure virtual functions with bodies

我们可以定义具有实体的纯虚函数:

#include <string>
 
class Animal // This Animal is an abstract base class
{
protected:
    std::string m_name;
 
public:
    Animal(const std::string& name)
        : m_name{ name }
    {
    }
 
    std::string getName() { return m_name; }
    virtual const char* speak() const = 0; // The = 0 means this function is pure virtual
    
    virtual ~Animal() = default;
};
 
const char* Animal::speak() const  // even though it has a body
{
    return "buzz";
}

在这种情况下，speak()仍然被认为是一个纯虚函数，因为“=0”(即使它已经被赋予了主体)，而Animal仍然被认为是一个抽象基类(因此不能实例化)。任何从Animal继承的类都需要为speak()提供自己的定义，否则它也将被视为一个抽象基类。

当为纯虚函数提供函数体时，必须单独提供(不内联)函数体。

当您希望基类为函数提供默认实现，但仍然强制任何派生类提供它们自己的实现时，此范例可能很有用。但是，如果派生类对基类提供的默认实现感到满意，则可以直接调用基类实现。例如:

#include <string>
#include <iostream>
 
class Animal // This Animal is an abstract base class
{
protected:
    std::string m_name;
 
public:
    Animal(const std::string& name)
        : m_name(name)
    {
    }
 
    const std::string& getName() const { return m_name; }
    virtual const char* speak() const = 0; // note that speak is a pure virtual function
    
    virtual ~Animal() = default;
};
 
const char* Animal::speak() const
{
    return "buzz"; // some default implementation
}
 
class Dragonfly: public Animal
{
 
public:
    Dragonfly(const std::string& name)
        : Animal{name}
    {
    }
 
    const char* speak() const override// this class is no longer abstract because we defined this function
    {
        return Animal::speak(); // use Animal's default implementation
    }
};
 
int main()
{
    Dragonfly dfly{"Sally"};
    std::cout << dfly.getName() << " says " << dfly.speak() << '\n';
 
    return 0;
}

# Interface classes

接口类没有成员变量，所有的函数都是纯虚函数。换句话说，类纯粹是一个定义，没有实际的实现。当您希望定义派生类必须实现的功能，而将派生类如何实现该功能的细节完全留给派生类时，接口非常有用。

接口类通常以i开头。下面是一个示例接口类:

class IErrorLog
{
public:
    virtual bool openLog(const char *filename) = 0;
    virtual bool closeLog() = 0;
 
    virtual bool writeError(const char *errorMessage) = 0;
 
    virtual ~IErrorLog() {} // make a virtual destructor in case we delete an IErrorLog pointer, so the proper derived destructor is called
};

任何继承IErrorLog的类都必须提供所有这三个函数的实现，以便被实例化。您可以派生一个名为FileErrorLog的类，其中openLog()打开磁盘上的一个文件，closeLog()关闭文件，而writeError()将消息写入该文件。您可以派生另一个名为ScreenErrorLog的类，其中openLog()和closeLog()不执行任何操作，而writeError()将消息打印在屏幕上的弹出消息框中。

现在，假设您需要编写一些使用错误日志的代码。如果你编写的代码中直接包含了FileErrorLog或ScreenErrorLog，那么你就只能使用这种错误日志了(至少不用重新编码你的程序)。例如，下面的函数有效地强制mySqrt()的调用者使用FileErrorLog，这可能是他们想要的，也可能不是。

#include <cmath> // for sqrt()
 
double mySqrt(double value, FileErrorLog &log)
{
    if (value < 0.0)
    {
        log.writeError("Tried to take square root of value less than 0");
        return 0.0;
    }
    else
    {
        return std::sqrt(value);
    }
}

现在调用者可以传入任何符合IErrorLog接口的类。如果他们想把错误转到一个文件中，他们可以传入一个FileErrorLog的实例。如果他们想让它转到屏幕上，他们可以传入一个ScreenErrorLog的实例。或者，如果他们想做一些你甚至没有想到的事情，比如在出现错误时发送电子邮件给某人，他们可以从IErrorLog(例如EmailErrorLog)派生一个新类，并使用它的一个实例!通过使用IErrorLog，您的函数变得更加独立和灵活。

不要忘记为接口类包含一个虚析构函数，以便在删除指向接口的指针时调用适当的派生析构函数。

接口类变得非常流行，因为它们易于使用、易于扩展和易于维护。事实上，一些现代语言，如Java和c#，已经添加了“interface”关键字，允许程序员直接定义接口类，而不必显式地将所有成员函数标记为抽象。此外，尽管Java(版本8之前)和c#不允许在普通类上使用多重继承，但它们可以让您任意多继承接口。因为接口没有数据和函数体，它们避免了许多传统的多重继承问题，同时仍然提供了很大的灵活性。

Pure virtual functions and the virtual table

抽象类仍然有虚表，因为如果您有一个指向抽象类的指针或引用，这些表仍然可以被使用。纯虚函数的虚表项通常要么包含一个空指针，要么指向一个打印错误的泛型函数(有时该函数被命名为__purecall)，如果没有提供覆盖。
