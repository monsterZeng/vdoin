# Composition

# Object composition

在现实生活中，复杂的物体通常是由更小、更简单的物体构成的。例如，一辆汽车是由金属框架、引擎、一些轮胎、传动装置、方向盘和许多其他部件组成的。个人电脑是由CPU、主板、一些内存等组成的，甚至你也是由更小的部件组成的:你有一个头、一个身体、一些腿、胳膊等等。这种从简单对象构建复杂对象的过程称为对象组合。、

广义地说，对象组合模型两个对象之间的“has-a”关系。汽车“有”变速器。你的电脑“有一个”中央处理器。你有一个心。复杂对象有时称为整体或父对象。较简单的对象通常称为部件、子对象或组件。

在c++中，您已经看到了结构和类可以具有各种类型的数据成员(如基本类型或其他类)。当我们用数据成员构建类时，本质上我们是用更简单的部分构造一个复杂的对象，这就是对象组合。出于这个原因，结构体和类有时被称为复合类型。

对象组合在c++上下文中很有用，因为它允许我们通过组合更简单、更容易管理的部分来创建复杂的类。这降低了复杂性，并允许我们更快地编写代码，减少错误，因为我们可以重用已经编写、测试和验证的代码。

# Types of object composition

对象组合有两种基本的子类型:组合和聚合。

关于术语的说明:术语“合成”通常指合成和聚合，而不仅仅是合成子类型。在本教程中，当我们指的是这两种类型时，我们将使用术语“对象合成”，当我们专门指合成子类型时，我们将使用“合成”。

# Composition

一个物体和一个部件必须有以下关系，才能成为一个组合:
  • The part (member) is part of the object (class).
  • The part (member) can only belong to one object (class) at a time
  • The part (member) has its existence managed by the object (class)
  • The part (member) does not know about the existence of the object (class)

一个很好的现实生活的例子是一个人的身体和心脏之间的关系。让我们更详细地研究这些问题。

组合关系是部分-整体关系，其中部分必须构成整个对象的一部分。例如，心脏是一个人身体的一部分。组合中的部分一次只能是一个对象的一部分。一颗心是一个人身体的一部分，不可能同时是另一个人身体的一部分。

在组合关系中，对象负责各个部分的存在。通常，这意味着在创建对象时创建部分，在销毁对象时销毁部分。但更广泛地说，它意味着对象以一种不需要对象用户参与的方式管理部件的生命周期。例如，当一个身体被创造出来时，心脏也被创造出来。当一个人的身体毁灭了，他的心也毁灭了。因此，合成有时被称为“死亡关系”。

最后，部分并不知道整体的存在。你的心以幸福的方式运作，没有意识到它是一个更大结构的一部分。我们称之为单向关系，因为身体知道心脏，而心脏不知道身体知道心脏。

请注意，组成与部件的可移植性无关。心脏可以从一个身体移植到另一个身体。然而，即使在移植之后，它仍然符合构成的要求(心脏现在属于接受者所有，除非再次移植，否则只能是接受者物体的一部分)。

我们无处不在的Fraction类就是一个很好的合成例子:

class Fraction
{
private:
	int m_numerator;
	int m_denominator;
 
public:
	Fraction(int numerator=0, int denominator=1):
		m_numerator{ numerator }, m_denominator{ denominator }
	{
		// We put reduce() in the constructor to ensure any fractions we make get reduced!
		// Since all of the overloaded operators create new Fractions, we can guarantee this will get called here
		reduce();
	}
};

这个类有两个数据成员:分子和分母。分子和分母是分数的一部分(包含在分数中)。它们一次不能属于一个以上的分数。分子和分母不知道它们是分数的一部分，它们只是整数。当创建Fraction实例时，分子和分母都创建了。当分数实例被销毁时，分子和分母也被销毁。

虽然对象组合模型具有类型关系(身体有心脏，分数有分母)，但我们可以更精确地说组合模型具有“部分-部分”关系(心是身体的一部分，分子是分数的一部分)。组合通常用于建模物理关系，其中一个对象在物理上包含在另一个对象中。

合成的各个部分可以是单数，也可以是乘法——例如，心脏是身体的单个部分，但身体包含10个手指(可以建模为一个数组)。

# Implementing compositions

组合是c++中最容易实现的关系类型之一。它们通常被创建为具有普通数据成员的结构体或类。因为这些数据成员直接作为结构/类的一部分存在，所以它们的生存期绑定到类实例本身的生存期。

需要进行动态分配或重新分配的组合可以使用指针数据成员来实现。在这种情况下，复合类应该自己负责进行所有必要的内存管理(而不是类的用户)。

一般来说，如果你能用组合来设计一个类，你应该用组合来设计一个类。使用组合设计的类简单、灵活、健壮(因为它们可以很好地自我清理)。

# More examples

许多游戏和模拟中都有在棋盘、地图或屏幕上移动的生物或物体。所有这些生物/物体都有一个共同点，那就是它们都有一个位置。在这个例子中，我们将创建一个生物类，它使用点类来保存生物的位置。

首先，让我们设计点类。我们的生物将生活在一个2d的世界里，所以我们的点类将有两个维度，X和y。我们假设世界是由离散的正方形组成的，所以这些维度总是整数。

#ifndef POINT2D_H
#define POINT2D_H
 
#include <iostream>
 
class Point2D
{
private:
    int m_x;
    int m_y;
 
public:
    // A default constructor
    Point2D()
        : m_x{ 0 }, m_y{ 0 }
    {
    }
 
    // A specific constructor
    Point2D(int x, int y)
        : m_x{ x }, m_y{ y }
    {
    }
 
    // An overloaded output operator
    friend std::ostream& operator<<(std::ostream& out, const Point2D &point)
    {
        out << '(' << point.m_x << ", " << point.m_y << ')';
        return out;
    }
 
    // Access functions
    void setPoint(int x, int y)
    {
        m_x = x;
        m_y = y;
    }
 
};
 
#endif

现在让我们设计我们的Creature。我们的Creature将会有一些属性:一个名称，它将是一个字符串，一个位置，它将是我们的Point2D类。

Creature.h

#ifndef CREATURE_H
#define CREATURE_H
 
#include <iostream>
#include <string>
#include "Point2D.h"
 
class Creature
{
private:
    std::string m_name;
    Point2D m_location;
 
public:
    Creature(const std::string &name, const Point2D &location)
        : m_name{ name }, m_location{ location }
    {
    }
 
    friend std::ostream& operator<<(std::ostream& out, const Creature &creature)
    {
        out << creature.m_name << " is at " << creature.m_location;
        return out;
    }
 
    void moveTo(int x, int y)
    {
        m_location.setPoint(x, y);
    }
};
#endif

And finally, Main.cpp:

#include <string>
#include <iostream>
#include "Creature.h"
#include "Point2D.h"
 
int main()
{
    std::cout << "Enter a name for your creature: ";
    std::string name;
    std::cin >> name;
    Creature creature{ name, { 4, 7 } };
	
    while (true)
    {
        // print the creature's name and location
        std::cout << creature << '\n';
 
        std::cout << "Enter new X location for creature (-1 to quit): ";
        int x{ 0 };
        std::cin >> x;
        if (x == -1)
            break;
 
        std::cout << "Enter new Y location for creature (-1 to quit): ";
        int y{ 0 };
        std::cin >> y;
        if (y == -1)
            break;
		
        creature.moveTo(x, y);
    }
 
    return 0;
}

# Variants on the composition theme

尽管大多数组合在创建就时直接创建他们的部分，在组合被破坏时直接破坏他们的部分，但也有一些变化的组合扭曲了这些规则。

例如：

  • 合成可能会推迟某些部件的创建，直到需要它们的时候。例如，字符串类可能不会创建一个动态字符数组，直到用户给字符串分配一些要保存的数据。
  • 组合可以选择使用已经给它的部件作为输入，而不是创建部件本身。
  • 组合可以将其部分的析构委托给其他对象(例如，垃圾收集例程)。

这里的关键点是，合成应该在不需要用户管理任何东西的情况下管理它的各个部分。

# Composition and subclasses

当涉及到对象组合时，新程序员经常问的一个问题是，“什么时候我应该使用一个子类，而不是直接实现一个特性?”例如，我们可以不使用Point2D类来实现生物的位置，而只向生物类添加2个整数，并在生物类中编写代码来处理定位。然而，使Point2D成为自己的类有许多好处:

  1. 每个单独的类可以保持相对简单和直接，专注于执行一个任务。这使得这些类更容易编写，也更容易理解，因为它们更有重点。例如，Point2D只关心与点相关的东西，这有助于保持简单。

  2. 每个子类都可以是自包含的，这使得它们可以重用。例如，我们可以在一个完全不同的应用程序中重用Point2D类。或者如果我们的生物曾经需要另一个点(例如，它试图到达的目的地)，我们可以简单地添加另一个Point2D成员变量。

  3. 父类可以让子类完成大部分繁重的工作，专注于协调子类之间的数据流。这有助于降低父对象的整体复杂性，因为它可以将任务委托给已经知道如何执行这些任务的子对象。例如，当我们移动我们的Creature时，它将该任务委托给Point类，它已经知道如何设置一个点。因此，Creature类不必担心这些东西是如何实现的。

一个很好的经验法则是，每个类都应该被构建来完成单一的任务。该任务应该是存储和操作某种类型的数据(例如Point2D, std::string)，或者协调子类(例如Creature)。

在我们的例子中，Creature不需要担心Points是如何实现的，或者名称是如何存储的。Creature的工作不是去了解那些私密的细节。Creature的工作是担心如何协调数据流，并确保每个子类知道它应该做什么。这取决于各个子类如何处理它。
